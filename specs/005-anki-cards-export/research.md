# Research: Export Book Annotations to Anki

**Branch**: `005-anki-cards-export` | **Date**: 2026-04-06

## Decision 1: HTTP Client for AnkiConnect

**Decision**: Use `urllib.request` from the Python standard library.

**Rationale**: AnkiConnect's interface is a single HTTP POST endpoint that
accepts a JSON body and returns a JSON response. `urllib.request` handles
this without any third-party package. All calls follow the same pattern:
POST to `http://localhost:8765` with `Content-Type: application/json` and
a body of `{"action": "...", "version": 6, "params": {...}}`.

**Alternatives considered**:
- `requests` / `httpx` — rejected; Constitution Principle V prohibits new
  dependencies when the stdlib can reasonably solve the problem. Both
  libraries would add install overhead with no functional benefit here.

---

## Decision 2: Duplicate Detection Strategy

**Decision**: Use `addNotes` with `options.allowDuplicate = false` (the
AnkiConnect default). The response is an array of note IDs (integers) for
successfully created cards and `null` for duplicates or errors. Count
nulls to determine the skipped total.

**Rationale**: Single round-trip per batch. AnkiConnect's built-in
duplicate checking on the "Basic" note type compares the Front field
within the target deck. No pre-query needed.

**AnkiConnect response contract**:
```json
{
  "result": [1234567890, null, 1234567891],
  "error": null
}
```
`null` at position `i` means note `i` was not added (duplicate or invalid).

**Alternatives considered**:
- `findNotes` query before each `addNote` — rejected; two round-trips per
  card and does not batch cleanly.
- `canAddNotes` — rejected; returns only a boolean array, not IDs; still
  requires a second call to actually add the notes.

---

## Decision 3: Input Type Detection

**Decision**: Detect input type by checking whether the given path is a
directory (→ `parse_annotation_dir`) or a file (→ `parse_notebook`).

**Rationale**: The two parsers accept distinct path types. A directory
unambiguously maps to the markdown parser; a file unambiguously maps to
the HTML parser. No `--format` flag is needed.

**Alternatives considered**:
- Explicit `--format html|md` flag — rejected; adds user burden with no
  benefit since the path type already carries the information.
- File extension detection (`.html`) — rejected; less reliable than
  directory/file distinction and couples to file naming conventions.

---

## Decision 4: Anki Card Front Format

**Decision**: Front = `{chapter} — {title}` when chapter is non-empty;
front = `{title}` when chapter is empty. If `title` is also empty or
whitespace, use the first 60 characters of `text` (ellipsised if longer).

**Rationale**: The chapter prefix gives the reader context about where the
annotation appears in the book. The title (auto-generated by the parser as
the first sentence, up to 7 words) is concise and sufficient as a prompt.

**Back field**: Always the full `text` of the annotation.

**Alternatives considered**:
- Include page/location number — rejected; adds noise without mnemonic
  value on the front of a flashcard.
- Use only text snippet — rejected; the title is already designed to be a
  concise label; using raw text on the front creates overly long prompts.

---

## Decision 5: Deck Naming

**Decision**: Default deck name = `book.title` from the first annotation.
Overridable with `--deck`. Deck created automatically via `createDeck`
action if it does not exist (AnkiConnect is idempotent on `createDeck`).

**Rationale**: The book title is always present (guaranteed by both
parsers). AnkiConnect's `createDeck` is safe to call even if the deck
already exists — it returns the existing deck ID without error.

---

## Decision 6: CLI Command Placement

**Decision**: Expose the feature as `otb anki export` (a new `anki` Click
group with an `export` subcommand), consistent with the existing `otb
kindle` and `otb md` groups.

**Rationale**: The codebase groups commands by source type (`kindle`,
`md`). An `anki` group is semantically parallel and leaves room for
future Anki-related subcommands (e.g., `otb anki sync`).

**Alternatives considered**:
- Top-level `otb anki-export` — rejected; breaks the `group/subcommand`
  pattern used everywhere else.

---

## Decision 7: MCP Tool

**Decision**: Register `anki_export` as an MCP tool in `mcp_server.py`,
delegating to the same service function used by the CLI command
(Constitution Principle VI).

**MCP tool signature**:
```python
def anki_export(
    path: str,
    deck: str = "",
    anki_url: str = "http://localhost:8765",
) -> dict[str, Any]
```
Returns `{"created": int, "skipped": int, "failed": int}`.

---

## Decision 8: New Module Structure

**Decision**: Add `src/otb/anki.py` containing:
1. `AnkiConnectError` — typed exception for API-level errors.
2. `AnkiClient` — thin wrapper around `urllib.request` for AnkiConnect
   calls (`create_deck`, `add_notes`).
3. `build_card` — pure function converting an `Annotation` to an Anki
   note dict.
4. `export_annotations` — service function orchestrating deck creation,
   card building, and batched note submission. Called by both CLI and MCP.

**Rationale**: Isolating Anki logic in its own module keeps `cli.py` and
`mcp_server.py` thin (Constitution Principle V: YAGNI, no duplication).
The service function pattern matches the existing codebase style.

## AnkiConnect API Reference (v6)

All calls: `POST http://localhost:8765`

### `createDeck`
```json
{"action": "createDeck", "version": 6, "params": {"deck": "My Deck"}}
```
Response: `{"result": 1234567890, "error": null}`

### `addNotes`
```json
{
  "action": "addNotes",
  "version": 6,
  "params": {
    "notes": [
      {
        "deckName": "My Deck",
        "modelName": "Basic",
        "fields": {"Front": "...", "Back": "..."},
        "options": {"allowDuplicate": false}
      }
    ]
  }
}
```
Response: `{"result": [1234567890, null], "error": null}`
`null` = duplicate or invalid note (not created).
