# Feature Specification: Kindle MCP Server

**Feature Branch**: `001-kindle-mcp-server`
**Created**: 2026-04-05
**Status**: Draft
**Input**: User description: "I need an MCP server that processes the highlights and notes exported from a Kindle app and exports them as a list of highlights and/or notes for further processing by the LLM. The MCP server should allow storing this highlights/notes in separate markdown files in a directory alongside the original Kindle export file. When MCP server parses the Kindle export the highlights should not have a title. The title should be generated by the LLM using smallest model (eg haiku)."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Parse Kindle Export into Highlights (Priority: P1)

An LLM agent provides the path to a Kindle HTML notebook export file and receives a
structured list of all highlights and notes. Highlights are returned without titles so
the LLM can decide whether and how to name them.

**Why this priority**: This is the foundational capability; all other stories depend on
having highlights available. Without parsing, there is nothing to title or save.

**Independent Test**: Can be fully tested by calling the parse tool with a Kindle HTML
fixture file and verifying a non-empty list of highlights is returned with correct book
metadata, chapter, page, location, and text — and that no title field is populated.

**Acceptance Scenarios**:

1. **Given** a valid Kindle HTML export file path, **When** the parse tool is called,
   **Then** a list of highlights is returned, each containing: book title, author,
   chapter, page, location, highlight text, and color (or absent if not available).
2. **Given** a valid Kindle HTML export, **When** highlights are returned,
   **Then** none of the highlights carry an auto-generated title.
3. **Given** a path to a non-existent or malformed file, **When** the parse tool is
   called, **Then** a descriptive error is returned and no partial results are emitted.

---

### User Story 2 - Generate a Title for a Highlight (Priority: P2)

An LLM agent provides the text of a single highlight and receives a short, meaningful
title generated by the smallest available Claude model (Haiku). The agent can then
attach this title to the highlight before saving or for its own use.

**Why this priority**: Title generation is the value-add step between parsing and
saving. It must work correctly before saving can produce well-named files.

**Independent Test**: Can be fully tested by calling the title-generation tool with a
sample highlight text and verifying the returned title is a non-empty string of
reasonable length (under 15 words) that captures the highlight's meaning.

**Acceptance Scenarios**:

1. **Given** a non-empty highlight text, **When** the generate-title tool is called,
   **Then** a concise title (under 15 words) is returned.
2. **Given** an empty or whitespace-only text, **When** the generate-title tool is
   called, **Then** a descriptive error is returned.
3. **Given** a highlight text in any language, **When** the generate-title tool is
   called, **Then** the returned title is in the same language as the highlight text.

---

### User Story 3 - Save Highlights as Markdown Files (Priority: P3)

An LLM agent provides a list of highlights (optionally with titles) and a target
directory path. The MCP server saves each highlight as an individual markdown file in
that directory. The directory is typically placed alongside the original Kindle export
file so the notes remain co-located with their source.

**Why this priority**: Saving is the persistence step. The feature delivers its full
value when highlights can be stored as structured markdown files for use in Obsidian
or other tools.

**Independent Test**: Can be fully tested by calling the save tool with a list of
highlights and a target directory, then verifying that one file per highlight is created
with correct content and frontmatter — and that the files can be re-read by the
existing markdown highlight parser.

**Acceptance Scenarios**:

1. **Given** a list of highlights and a writable directory path, **When** the save tool
   is called, **Then** one markdown file per highlight is created in that directory.
2. **Given** highlights with titles, **When** saved, **Then** each file's frontmatter
   includes the title, book, author, chapter, page, location, and color (if present).
3. **Given** highlights without titles, **When** saved, **Then** each file is saved
   without a title field in the frontmatter (no auto-generation at save time).
4. **Given** a directory that does not yet exist, **When** the save tool is called,
   **Then** the directory is created and files are saved successfully.
5. **Given** a non-writable directory, **When** the save tool is called, **Then** a
   descriptive error is returned and no partial files are written.

---

### Edge Cases

- What happens when the Kindle export contains no highlights? The parse tool returns an
  empty list (not an error).
- What happens when two highlights have identical text? Each is saved as a separate file
  with a unique name (e.g., by appending an index).
- What happens when a highlight text is very long? The generated title is still capped
  at under 15 words; the full text is preserved in the file body.
- What happens when saving is interrupted mid-batch? Already-written files remain; the
  error identifies which highlights were not saved.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The MCP server MUST expose a tool that accepts a file path to a Kindle
  HTML export and returns a structured list of highlights and notes.
- **FR-002**: Highlights returned by the parse tool MUST NOT include auto-generated
  titles; the title field MUST be absent or empty.
- **FR-003**: The MCP server MUST expose a tool that accepts a highlight's text and
  returns a short title generated by the smallest available Claude model (Haiku).
- **FR-004**: The title-generation tool MUST use the Claude Haiku model specifically
  (not a larger model) to keep costs low.
- **FR-005**: The MCP server MUST expose a tool that accepts a list of highlights and
  a target directory path, and saves each highlight as an individual markdown file.
- **FR-006**: Each saved markdown file MUST contain highlight metadata (book title,
  author, chapter, page, location, color) in a YAML-like frontmatter block.
- **FR-007**: The saved markdown file format MUST be compatible with the existing
  markdown highlight parser so files can be round-tripped.
- **FR-008**: The MCP server MUST operate via stdio transport.
- **FR-009**: All three tools MUST be usable independently; no implicit coupling
  between parse → title → save is enforced by the server.

### Key Entities

- **Highlight**: The unit of content returned and stored. Attributes: book (title,
  author), chapter, page, location, highlight text, color (optional), title
  (optional — absent until the LLM generates one).
- **Book**: Identified by title and author; groups highlights that come from the same
  source work.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An LLM agent can parse a Kindle export containing 100+ highlights and
  receive the full list in a single tool call within 5 seconds.
- **SC-002**: Generated titles are under 15 words and judged relevant to the highlight
  text in 90%+ of cases during manual spot-checks.
- **SC-003**: Saved markdown files can be re-parsed by the existing highlight parser
  without errors or data loss (round-trip fidelity).
- **SC-004**: The MCP server starts and responds to tool calls from any
  MCP-compatible client (e.g., Claude Desktop) without additional configuration
  beyond pointing to the server binary/command.

## Assumptions

- The Kindle export is a single HTML file in the format currently produced by the
  Kindle app (as handled by the existing HTML parser).
- The target directory for saving markdown files defaults to a subdirectory named
  after the Kindle export file (without extension) in the same parent directory;
  callers may override this by passing an explicit path.
- Markdown filenames are derived from the highlight's sequential index within the
  book (e.g., `001 - ...`, `002 - ...`); if a title is present, a sanitized version
  of the title is appended after the index.
- The smallest Claude model currently available is Claude Haiku; the title-generation
  tool references this model by its stable ID so it can be updated when a newer Haiku
  is released.
- The LLM agent orchestrates the full workflow (parse → generate titles → save); the
  MCP server does not chain these steps automatically.
- Color information is available from HTML exports but absent from markdown-sourced
  highlights; the server preserves whatever color data the parser provides.
